# Design: cmdk palette v1.5A - Selection commands without secondary pickers

## Goal

Add the first selection-aware command slice to the global cmdk palette.
When the user has selected assets, cmdk will expose the common bulk
asset verbs that are already present in the selection toolbar.

This is intentionally v1.5A, not the whole v1.5 roadmap. It adds
explicit selection commands and reuses existing page actions, modals,
SDK calls, and v1.4 inline confirmation. It does not add in-palette
secondary pickers. The modal target-space picker follow-up is specified in
`docs/superpowers/specs/2026-04-28-cmdk-add-selected-to-space-design.md`.

## Prior roadmap

The v1.3 and v1.4 cmdk designs deferred selection verbs to v1.5:

- Add selected to album
- Add selected to space
- Favorite selected
- Archive selected
- Delete selected

They also sketched multi-step palette state for "Add to album...", where
the palette would switch from command search into an album picker. That
multi-step picker is still the right larger direction, but it is not
needed for the first useful slice.

## Scope

In scope:

- Selection-aware `CommandContext` data.
- Explicit cmdk commands for:
  - Add selected to album...
  - Add selected to this space
  - Add selected to favorites
  - Archive selected
  - Delete selected
- Existing `AssetAddToAlbumModal` for album picking.
- Current-space add only on writable space pages while the page is in
  its existing asset-picking mode.
- Existing v1.4 inline confirm for delete.
- Focused unit/component tests and one route-level integration test per
  new page-wiring behavior.

Out of scope:

- In-palette album or space secondary pickers.
- Bridging every `CommandPaletteDefaultProvider` action into global cmdk.
- New toolbar layout or a new visible cmdk launcher in selection bars.
- Remove from favorites / unarchive selected commands. Those can follow
  after the add/archive-only verbs are stable.
- Tagging, metadata, stack, rotate, live-photo link, visibility, and job
  commands.

## In-palette secondary pickers

An in-palette secondary picker means the cmdk panel changes modes after a
command is chosen. Example: the user selects "Add selected to album...",
then the same palette turns into an album search view with a back affordance,
album rows, and an album-specific empty/loading state. Escape would return
to the command list, and Enter would add to the highlighted album.

v1.5A does not build that mode system. "Add selected to album..." closes
or yields the palette to the existing `AssetAddToAlbumModal`, which already
handles album selection and add semantics.

## Command inventory

| id                                   | Label                      | Available when                                                                                                                          | Handler                                                                                             | Selection clearing               |
| ------------------------------------ | -------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------- | -------------------------------- |
| `cmd:selection_add_to_album`         | Add selected to album...   | Selection exists and the page registered `canAddToAlbum`                                                                                | `modalManager.show(AssetAddToAlbumModal, { assetIds })`                                             | No explicit command clear        |
| `cmd:selection_add_to_space`         | Add selected to space...   | Selection exists and the page registered `canAddToSpace`                                                                                | `modalManager.show(AssetAddToSpaceModal, { assetIds })`                                             | No explicit command clear        |
| `cmd:selection_add_to_current_space` | Add selected to this space | Space page, user can write, page is in existing `select-assets` mode, selection count is 1..`MAX_SPACE_ASSETS_PER_REQUEST`              | Delegate to the page's existing add-to-space handler                                                | Page helper clears on success    |
| `cmd:selection_favorite`             | Add selected to favorites  | Selection exists, all selected assets are user-owned, page registered `onFavorite`, at least one selected asset is not already favorite | `updateAssets({ ids, isFavorite: true })`, mutate local selected asset refs, call page `onFavorite` | `clearSelection()` after success |
| `cmd:selection_archive`              | Archive selected           | Selection exists, all selected assets are user-owned, page registered `onArchive`, at least one selected asset is not already archived  | `updateAssets({ ids, visibility: Archive })`, call page `onArchive`                                 | `clearSelection()` after success |
| `cmd:selection_delete`               | Delete selected            | Selection exists, all selected assets are user-owned, page registered `onDelete`                                                        | v1.4 destructive two-Enter confirm, then delete selected assets                                     | `clearSelection()` after success |

The current-space command is deliberately not a general space picker. It is
only "Add selected to this space". On the current codebase it is scoped to
the space page's `viewMode === 'select-assets'`; in normal space `view` mode,
selected assets are already inside the space, so showing "Add selected to
this space" would be a confusing duplicate/no-op.

## Context design

Extend `CommandContext` with a nullable `selection` subcontext:

```typescript
export interface SelectionCommandContext {
  assets: TimelineAsset[];
  selectedAssetIds: string[];
  ownedAssets: TimelineAsset[];
  ownedSelectedAssetIds: string[];
  canAddToAlbum: boolean;
  isAllUserOwned: boolean;
  isAllFavorite: boolean;
  isAllArchived: boolean;
  isAllTrashed: boolean;
  clearSelection: () => void;
  onFavorite?: OnFavorite;
  onArchive?: OnArchive;
  onDelete?: OnDelete;
  onUndoDelete?: OnUndoDelete;
  addSelectedToCurrentSpace?: () => Promise<boolean>;
}

export interface CommandContext {
  routeId: string | null;
  params: Record<string, string>;
  album: AlbumContext | null;
  space: SpaceContext | null;
  selection: SelectionCommandContext | null;
  userId: string | null;
  isAdmin: boolean;
}
```

Add a `registerSelectionContext` helper beside `registerAlbumContext` and
`registerSpaceContext`. Pages call it with thunks that read
the page's active selection manager and page-local callbacks. The helper
stores a reactive selection context while the page is mounted and clears it
on unmount.

The context manager should return `selection: null` when the selected asset
list is empty. Handlers must still guard at activation time because the
selection can change while the palette is open.

`registerSelectionContext` should also store the current route id at
registration time. `getContext()` returns the stored selection only while the
current route id still matches that registered route. This mirrors the v1.4
album/space stale-context guard and prevents selection commands from leaking
onto another page if navigation and component cleanup interleave.

Expected helper shape:

```typescript
type RegisterSelectionContextOptions = {
  getAssets: () => TimelineAsset[];
  clearSelection: () => void;
  canAddToAlbum?: () => boolean;
  getOnFavorite?: () => OnFavorite | undefined;
  getOnArchive?: () => OnArchive | undefined;
  getOnDelete?: () => OnDelete | undefined;
  getOnUndoDelete?: () => OnUndoDelete | undefined;
  getAddSelectedToCurrentSpace?: () => (() => Promise<boolean>) | undefined;
};
```

`getAssets`, `clearSelection`, and the callback getters make the command
implementation independent of the global `assetMultiSelectManager` singleton.
That matters because some pages, such as album asset-picking mode, use a
different multi-select manager. Those modes should opt in only when the
selected assets represent the same semantic selection as the visible toolbar.

The public `SelectionCommandContext` exposes resolved callbacks
(`onFavorite`, `onArchive`, `onDelete`, `onUndoDelete`) for command handlers.
`registerSelectionContext` accepts callback getters so page mode changes can
hide or expose actions without a route remount. A page with static behavior
can pass `getOnFavorite: () => handleFavorite`.

`getContext()` should build the `selection` snapshot from the registered
thunks each time it is read, not keep a one-time copy of derived flags. This
lets command availability follow page-mode changes without remounting the
route, such as a space page moving between `view` and `select-assets`.
`selectedAssetIds` should be unique and in the same order as `assets`.

## Page wiring

Wire selection context only on pages that can provide correct local update
callbacks. Do not expose a cmdk command on a page unless the same behavior is
already available in that page's selection UI or the page has an explicit
page-local callback that keeps the visible state correct.

Expected first pages:

| Page/state                    | Selection source             | Registered capabilities                                                                                                                                           |
| ----------------------------- | ---------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Photos page                   | `assetMultiSelectManager`    | Add to album, favorite, archive, delete, undo delete                                                                                                              |
| Album detail, view mode       | `assetMultiSelectManager`    | Add to album, favorite, archive, delete, undo delete                                                                                                              |
| Album detail, add-assets mode | `timelineMultiSelectManager` | No v1.5A selection context. That selection belongs to the album add flow, not global bulk commands.                                                               |
| Space detail, view mode       | `assetMultiSelectManager`    | Favorite and archive only. Do not register add-to-album or delete unless the page selection UI grows those behaviors.                                             |
| Space detail, add-assets mode | `assetMultiSelectManager`    | Add selected to this space only, and only when `space.canWrite`.                                                                                                  |
| Search results page           | `assetMultiSelectManager`    | Add to album, favorite, archive with correct `searchResultAssets` update/removal, delete, undo delete                                                             |
| Tags page                     | `assetMultiSelectManager`    | Add to album, favorite, archive, delete, undo delete                                                                                                              |
| Archive page                  | `assetMultiSelectManager`    | Add to album, add to favorites, delete. "Archive selected" remains hidden because archive-page selections are already archived; "Unarchive selected" is deferred. |

Each page registers only the callbacks it can support correctly. A command
that needs a missing callback stays hidden instead of doing an SDK call that
leaves the visible page stale.

Callbacks must be registered through the dynamic getters above when page mode
affects action availability. The space page depends on this: normal `view`
mode can expose favorite/archive for assets already in the space, while
`select-assets` mode exposes only add-to-current-space for candidate assets.

Do not wire v1.5A selection context on other selection pages yet, including
favorites, people, folders, map, trash, locked, utilities, and memory pages.
Several of those pages have related toolbar actions, but they also introduce
different semantics such as remove-from-favorites, permanent locked delete,
restore-from-trash, or person-specific refresh. Those are follow-up page
contexts, not hidden scope inside v1.5A.

The space page gets one extra page-local capability:

- When `viewMode === 'select-assets'` and `space.canWrite`, provide
  `addSelectedToCurrentSpace`.
- Otherwise omit `addSelectedToCurrentSpace`, which hides the command.

The handler should reuse the same limit check and add flow as the current
plus button: no selected assets, too many selected assets, SDK failures, and
the `SpaceAddAssets` event all behave the same way.

Do not rely on `eventManager.emit()` to await async listeners; the base event
manager intentionally fire-and-forgets callbacks. Prefer extracting an
awaited page-local `addSelectedAssetsToCurrentSpace()` helper that both the
plus button and cmdk callback call. That helper should perform the SDK add,
emit `SpaceAddAssets` for external subscribers, refresh the current space,
load activities, clear selection, return to `view` mode, and return `true`.
Validation stops and handled errors return `false`.

## Handler details

### Add selected to album

Use the existing modal:

```typescript
modalManager.show(AssetAddToAlbumModal, {
  assetIds: ctx.selection.selectedAssetIds,
});
```

Availability requires `ctx.selection.canAddToAlbum`. The command is
non-destructive. Canceling the modal changes nothing. A successful add
follows the modal/service's existing toast and event behavior. Selection
is not explicitly cleared by the command because this mirrors the current
selection toolbar action. Page-level `AlbumAddAssets` listeners may still
clear selection or remove rows when they already do so for the toolbar path,
such as search results with `isNotInAlbum` filters.

### Add selected to this space

Use the page-provided `addSelectedToCurrentSpace` callback instead of
duplicating space-page state in `command-items.ts`. The callback returns a
boolean success result:

- `true`: add completed and the page performed its normal refresh/clear.
- `false`: no-op, validation stop, cancel, or handled error; cmdk does not
  clear anything itself.

Availability must require:

- Current route has `ctx.space`.
- `ctx.space.canWrite` is true.
- `ctx.selection` exists.
- `ctx.selection.addSelectedToCurrentSpace` exists.
- `selectedAssetIds.length > 0`.
- `selectedAssetIds.length <= MAX_SPACE_ASSETS_PER_REQUEST`.

### Add selected to favorites

The command should match existing toolbar ownership rules: it is available
only when all selected assets are user-owned. Mixed owner selections should
not silently apply to only the owned subset.

Availability requires `onFavorite`; pages that cannot update visible state
do not get the command. Filter out assets that are already favorite before
calling the SDK. If the filtered id list is empty, hide the command. On
success:

- Mutate the selected asset refs to `isFavorite = true`.
- Call `onFavorite(ids, true)`.
- Show the same count toast as the toolbar action.
- Call `clearSelection()`.

If the filtered id list becomes empty at activation time, no-op without SDK
call, toast, or selection clear. On SDK failure, show the existing error
toast and keep selection.

### Archive selected

Same ownership and no-op rules as favorite. Availability requires
`onArchive`; pages that cannot update visible state do not get the command.
Use direct `updateAssets` or fix `archiveAssets()` before reusing it. The
current `archiveAssets()` helper returns ids even after catching an SDK
failure, so a command must not rely on that return value for success
semantics unless the helper is changed first.

On success:

- Call `onArchive(ids, AssetVisibility.Archive)`.
- Call `clearSelection()`.

If the filtered id list becomes empty at activation time, no-op without SDK
call, toast, or selection clear. On failure, show the existing error toast
and keep selection.

### Delete selected

Mark the command `destructive: true` so the existing v1.4 inline-confirm
state machine handles first Enter vs second Enter, Escape, query changes,
and stale pending confirm cancellation.

The actual delete call should return success/failure. Either update
`deleteAssets()` in `web/src/lib/utils/actions.ts` to return `true` on SDK
success and `false` after handled errors, or add a command-specific helper
with that contract. Existing callers may ignore the returned boolean.

Preserve the existing permanent-delete safety behavior: v1.4 inline confirm
is required for every palette delete, and if the resolved delete is a
force-delete while `$showDeleteModal` is true, show `AssetDeleteConfirmModal`
before the SDK call. Canceling that modal returns `false`, keeps selection,
and does not call page update callbacks.

Force-delete rules:

- Force when trash is disabled.
- Force when any selected asset is already trashed.
- Otherwise move to trash.

On success:

- Call `onDelete(ids)` exactly once, either inside the helper or directly in
  the command handler.
- Wire undo through `onUndoDelete` only for non-force deletes.
- Call `clearSelection()`.

On failure, keep selection and do not call page update callbacks.

## Error handling and stale-state rules

- Command availability is an affordance, not the only guard. Every handler
  rechecks `ctx.selection` and required ids/callbacks at activation time.
- `clearSelection` comes from the registered selection context. Command
  handlers should not import and clear the global `assetMultiSelectManager`
  directly.
- SDK failures show the existing localized error toast and do not clear
  selection.
- Modal cancel is not an error and does not clear selection.
- Mixed ownership hides favorite/archive/delete for parity with existing
  toolbar behavior.
- Add-to-album can use all selected ids only on pages that set
  `canAddToAlbum`.
- Add-to-current-space can use all selected ids in space add-assets mode.
- Add-to-current-space is hidden in normal space view mode to avoid adding
  already-in-space assets to the same space.
- Add-to-current-space is hidden for viewer/read-only space members.
- Add-to-current-space is hidden over `MAX_SPACE_ASSETS_PER_REQUEST`.
- Delete is hidden on the space page unless the page registers a real
  asset-delete callback. The current space toolbar removes assets from a
  space; it does not delete them from the library.
- If navigation or page unmount clears the registered context while cmdk is
  open, activation becomes a no-op.
- If selection changes while cmdk is open, activation uses the live context
  snapshot, not the stale row text.
- If page mode changes without route navigation, availability follows the
  latest registered thunks. For example, space `view` mode hides
  add-to-current-space, while `select-assets` mode can expose it.
- Non-wired pages with selected assets produce `ctx.selection === null`, so
  no selection command appears there in v1.5A.

## TDD requirement

Implementation must use TDD. No production code for this feature is written
until a focused failing test exists and has been run.

Use vertical red-green-refactor slices:

1. Write one failing test for one behavior.
2. Run the focused test and confirm it fails for the expected reason.
3. Write the smallest implementation that makes that test pass.
4. Run the focused test again and confirm it passes.
5. Run the relevant neighboring tests before moving to the next behavior.
6. Refactor only while tests are green.

Do not write all tests first and then implement the whole feature. Do not
write implementation first and backfill tests. Each implementation commit
should have its own red/green evidence in the working notes or PR body.

## Edge cases

- Empty selection while cmdk is open: command handler no-ops.
- Selection changes after the result row was rendered: activation uses the
  current selected ids.
- Asset already favorite: favorite command omits it from the SDK update.
- All selected assets already favorite: favorite command is hidden.
- Asset already archived: archive command omits it from the SDK update.
- All selected assets already archived: archive command is hidden.
- Mixed owned/shared selection: favorite/archive/delete are hidden.
- Non-owned selection: favorite/archive/delete are hidden.
- Add-to-album modal canceled: selection remains.
- Add-to-album service failure: modal/service handles toast; command does not
  clear selection itself.
- Add-to-album success on pages with existing `AlbumAddAssets` listeners may
  clear selection or remove rows through those listeners.
- Space view selection: add-to-album and delete are hidden because the current
  toolbar does not expose those actions there.
- Space mode changes while cmdk is open: activation uses the latest context;
  add-to-current-space no-ops if the page left `select-assets` mode.
- Space viewer opens cmdk while assets are selected: add-to-current-space is
  hidden.
- Space editor selects more than `MAX_SPACE_ASSETS_PER_REQUEST`: add-to-current
  space is hidden.
- Space add-assets command callback returns false after validation failure or
  handled SDK failure: cmdk does not clear selection itself.
- Favorites/trash/locked/non-wired pages with selected assets: no v1.5A
  selection commands appear.
- Favorite/archive rows become no-op if an external update makes every
  selected asset already favorite/archived before activation.
- Duplicate selected ids from a future custom selection source are de-duped
  before SDK calls.
- Delete confirm times out: no delete call.
- Permanent delete modal canceled after inline confirm: no delete call.
- Force delete due to already-trashed selection or trash disabled: no undo
  button is wired.
- Delete SDK failure: no clear, no page callback.
- Non-force delete with undo callback: undo restores through the existing
  toast action.
- Navigation while pending delete confirm: existing v1.4 close-on-navigation
  clears pending confirm and page selection manager clears selection.
