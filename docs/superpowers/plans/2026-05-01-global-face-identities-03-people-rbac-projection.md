# Global Face Identities: People RBAC Projection Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:test-driven-development for every code task. Use superpowers:verification-before-completion before claiming this phase is done. This is phase 3 of the global face identities roadmap and depends on phases 1 and 2.

**Goal:** Make `/people` return one row per accessible Face Identity across the viewer's own library and timeline-enabled shared spaces, without exposing inaccessible metadata, thumbnails, counts, spaces, or raw identity ids.

**Architecture:** Add an access-scoped people resolver to `FaceIdentityRepository`. It applies the viewer's asset scope before grouping by identity, pages identity ids first, then hydrates only the selected page with scoped profiles, accessible counts, accessible thumbnails, and a scoped primary profile for navigation.

**Tech Stack:** NestJS, Kysely, `@immich/sql-tools`, Vitest small and medium tests, generated OpenAPI SDK, Svelte 5 route/component tests.

---

## Scope

This phase owns:

- Identity-grouped `GET /people?withSharedSpaces=true`.
- Accessible display metadata selection for `/people`.
- Accessible counts and thumbnails for `/people`.
- `primaryProfile` route targets for personal and space people.
- `/people` web route loading and navigation.
- `/people` RBAC matrix assertions that were deferred from phases 1 and 2.

This phase does not own:

- Filter-panel scoped person tokens.
- Global search people rows and previews.
- `searchPerson({ withSharedSpaces: true })`; that is implemented in phase 4 so the same search endpoint can serve both `/people` search and global search.
- Manual same-person repair or detach endpoints.

- [ ] **Step 3: Implement the identity page query**

Query shape:

```text
accessible asset scope
  -> visible asset_face rows
  -> face_identity_face
  -> group by face_identity_face.identityId
  -> order by best accessible profile name then identity id
  -> limit size + 1 offset (page - 1) * size
```

Accessible asset scope includes:

- assets owned by the user that are eligible for the personal timeline;
- assets in shared spaces where `shared_space_member.userId = userId` and `showInTimeline = true`;
- assets from linked space libraries that the user can access through those spaces.

Rules:

- Apply asset access before identity grouping.
- Exclude deleted or invisible faces.
- Apply `withHidden` only to the scoped profiles used for display, not to raw identity existence.
- Fetch `size + 1` rows so `hasNextPage` does not require an unbounded total count.
- Compute `total` and `hidden` from the same accessible identity scope. If the implementation uses `COUNT(DISTINCT ...)`, add generated SQL coverage and carry the query into phase 5 `EXPLAIN` review before shipping.
- Compute `numberOfAssets` from distinct accessible asset ids.

- [ ] **Step 4: Implement hydration and display priority**

For each identity on the page, hydrate accessible scoped profiles only.

Display priority:

1. viewer's own `person` profile;
2. viewer's alias on an accessible `shared_space_person`;
3. best accessible `shared_space_person`;
4. unnamed fallback.

Primary profile priority:

1. viewer's own `person` profile when present;
2. accessible `shared_space_person` with the best display metadata;
3. accessible `shared_space_person` with the newest accessible face.

Thumbnail rules:

- own `person.thumbnailPath` may be used only for the viewer's own profile;
- space-primary thumbnails must come from an asset visible in that space;
- never use a private source asset outside the viewer's accessible scope.

Output rules:

- `id` remains the primary profile id for backward compatibility.
- `primaryProfile.type = 'user-person'` yields `filterId = person:<personId>`.
- `primaryProfile.type = 'space-person'` yields `filterId = space-person:<sharedSpacePersonId>`.
- `isFavorite`, `color`, and private `isHidden` come only from the viewer's own `person`; for space-primary rows, omit optional favorite/color and use space visibility only when the viewer can manage or see that space profile state.
