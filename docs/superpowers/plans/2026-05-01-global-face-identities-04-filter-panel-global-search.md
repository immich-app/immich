# Global Face Identities: Filter Panel And Global Search Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:test-driven-development for every code task. Use superpowers:verification-before-completion before claiming this phase is done. This is phase 4 of the global face identities roadmap and depends on phases 1-3.

**Goal:** Use scoped identity filter tokens in global/timeline filters and global search so person suggestions, person filters, `/people` search, and global search dedupe across accessible spaces without exposing raw identity ids.

**Architecture:** Public APIs continue to accept legacy bare person UUIDs, but identity-grouped suggestions return scoped tokens such as `person:<id>` and `space-person:<id>`. The server resolves those tokens under the current viewer's access scope before filtering assets or ranking search results; single-space routes continue to use `spacePersonIds`.

**Tech Stack:** NestJS, Kysely, `@immich/sql-tools`, Vitest small and medium tests, generated OpenAPI SDK, Svelte 5 utility/component/manager tests.

---

## Scope

This phase owns:

- Scoped person tokens in filter DTOs and search suggestion DTOs.
- Server-side token resolution and access validation.
- Identity-backed global/timeline people filter suggestions.
- Asset filtering by accessible identity.
- `/people` search with `withSharedSpaces=true`.
- Global search person rows, navigation, and previews.
- Filter/global-search RBAC assertions from the permission matrix.

This phase does not own:

- Manual same-person repair and detach endpoints.
- Performance release sign-off for all query plans, except the local query-shape checks for filters and search introduced here.
- Any request-time vector matching.

## TDD Rules

- Add token-resolution and RBAC tests before changing filter/search queries.
- Treat malformed tokens as `BadRequestException`.
- Treat inaccessible but well-formed scoped tokens as an empty resolved identity set so asset results are empty rather than broadened.
- Keep legacy bare UUID support for existing personal filters.
- Keep single-space routes on `spacePersonIds`; do not send scoped tokens to single-space asset search.
- Never rank global search or people search by inaccessible names, aliases, spaces, thumbnails, or counts.
- Do not expose raw `face_identity.id` through DTOs, OpenAPI, SDK, web state, URLs, logs, or rendered output.

- [ ] **Step 4: Implement token parsing and resolution**

Add:

```ts
type ScopedPersonTokenResolution = {
  identityIds: string[];
  legacyPersonIds: string[];
  legacySpacePersonIds: string[];
  hasInaccessibleToken: boolean;
};

resolveScopedPersonTokens(input: {
  userId: string;
  tokens: string[];
  scope: { withSharedSpaces?: boolean; spaceId?: string; timelineSpaceIds?: string[] };
}): Promise<ScopedPersonTokenResolution>;
```

Resolution rules:

- `person:<personId>` requires the user to own that `person`; shared-space access must use a `space-person:<sharedSpacePersonId>` token.
- `space-person:<sharedSpacePersonId>` requires the user to be a member of that space.
- in global/timeline contexts, `space-person:<id>` must also belong to a timeline-enabled space included in the current request scope;
- in single-space contexts, scoped tokens are rejected and callers must use `spacePersonIds`;
- bare UUIDs remain legacy personal person filters and are returned in `legacyPersonIds`; existing repository access checks must still restrict them to the caller's personal scope.
- duplicate tokens are deduped.
- well-formed but inaccessible tokens set `hasInaccessibleToken = true`.
- malformed tokens are rejected by DTO validation before repository resolution.
