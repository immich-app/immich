# Storage Migration E2E Test Script — Design

## Overview

A two-part test harness for end-to-end validation of storage migration between disk and S3 (MinIO):

1. **`e2e/storage-migration.sh`** — Bash orchestrator managing Docker infrastructure (compose up/down, MinIO setup, server restarts with different env vars)
2. **`e2e/src/storage-migration.ts`** — TypeScript test logic (upload assets, call migration API, validate results)

The bash script calls the TypeScript test at different phases, passing the current phase as an argument.

## Architecture

```
storage-migration.sh
  |
  +-- Phase 1: SETUP
  |   +-- docker compose up (immich + postgres + redis + minio)
  |   +-- create MinIO bucket via mc CLI
  |   +-- run TS: upload test assets (disk mode)
  |
  +-- Phase 2: MIGRATE TO S3
  |   +-- restart immich-server with IMMICH_STORAGE_BACKEND=s3 env vars
  |   +-- run TS: trigger migration, poll status, validate (API + S3 + DB)
  |
  +-- Phase 3: MIGRATE BACK TO DISK
  |   +-- restart immich-server with IMMICH_STORAGE_BACKEND=disk
  |   +-- run TS: trigger migration, poll status, validate (API + disk + DB)
  |
  +-- Phase 4: CLEANUP (with --cleanup flag)
      +-- docker compose down -v
```

## Infrastructure

The bash script extends the existing `e2e/docker-compose.yml` with an override file `e2e/docker-compose.storage-migration.yml` that adds:

- **MinIO** container (port 9000/9001) with default credentials
- Volume mounts shared between immich-server and the test runner

## Test Assets

Minimal valid files created as byte buffers in TypeScript — no external dependencies. Only enough bytes to be valid file headers for each format.

| Category         | What gets uploaded/created                  | How                                          |
| ---------------- | ------------------------------------------- | -------------------------------------------- |
| Original image   | Minimal valid JPEG (~107 bytes)             | Upload via `POST /assets`                    |
| Original video   | Minimal valid MP4 (smallest ftyp+moov+mdat) | Upload via `POST /assets`                    |
| Thumbnail        | Generated by server after upload            | Wait for job completion                      |
| Preview          | Generated by server after upload            | Wait for job completion                      |
| Encoded video    | Generated by server after upload            | Wait for job completion                      |
| Sidecar          | XMP file uploaded alongside asset           | Upload via sidecar field                     |
| Person thumbnail | Face detection creates this                 | Insert directly into DB (ML disabled in e2e) |
| Profile image    | User avatar                                 | Upload via `PUT /users/profile-image`        |

Since ML is disabled in e2e, **person thumbnails** will be seeded directly into the DB and filesystem. Thumbnails and previews are generated by the server without ML (image resizing only).

## Validation Strategy

After each migration direction, the TypeScript test validates:

1. **API checks** — `GET /assets/:id/thumbnail`, `GET /assets/:id/original` return 200 with correct content
2. **Backend checks:**
   - After toS3: files exist in MinIO bucket, files absent from disk, DB paths are relative
   - After toDisk: files exist on disk, files absent from MinIO, DB paths are absolute
3. **DB checks** — Query `asset.originalPath`, `asset_file.path`, `person.thumbnailPath`, `user.profileImagePath` and verify path format (relative vs absolute)
4. **Migration log** — Query `storage_migration_log` table to verify entries were created with correct direction and batch ID

## Script Flags

```
./storage-migration.sh [options]
  --cleanup       Tear down all infrastructure after run
  --no-cleanup    Keep infrastructure running (default)
  --phase <n>     Run only a specific phase (for debugging)
  --verbose       Show docker/server logs inline
```
