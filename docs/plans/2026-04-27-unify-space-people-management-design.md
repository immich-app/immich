# Unify Space People Management Design

## Goal

Reuse the same frontend people-management building blocks for global people and space people while exposing currently unused space-person capabilities in the frontend. Birth date editing is in scope. Alias display and editing are also in scope, using the existing space-person alias endpoints.

## Context

Global people management and space people management currently use separate frontend implementations for similar workflows:

- Global people use `web/src/routes/(user)/people/+page.svelte`, `people-card.svelte`, `people-infinite-scroll.svelte`, and `manage-people-visibility.svelte`.
- Space people use `web/src/routes/(user)/spaces/[spaceId]/people/+page.svelte` and `web/src/lib/components/spaces/manage-space-people-visibility.svelte`.
- Space person detail uses `web/src/routes/(user)/spaces/[spaceId]/people/[personId]/+page.svelte`.

The backend already supports the required space-person data and operations:

- `SharedSpacePersonResponseDto` includes `birthDate` and `alias`.
- `SharedSpacePersonUpdateDto` accepts `birthDate`.
- Shared-space controller exposes `updateSpacePerson`, `setSpacePersonAlias`, and `deleteSpacePersonAlias`.

The implementation should avoid backend runtime/API changes, SDK shape changes, route changes, and file moves. This fork rebases on upstream, so the design prioritizes additive frontend extraction, minimal backend test coverage, and small adapter edits in existing files.

## Architecture

Use a compatibility-first extraction. Add new shared people components under `web/src/lib/components/people/`, and keep existing route and route-specific component paths in place as adapters.

The shared components should own repeated UI structure and interaction behavior:

- People grid layout and pagination sentinel.
- Person tile shell: thumbnail, badges, hover menu area, footer slot, read-only label/input slot.
- Visibility-management modal behavior: local hidden-state overrides, preset toggle cycle, save lifecycle, pagination, loading state.

The existing global and space pages should continue to own route-specific concerns:

- Data loading and pagination API calls.
- Search and scroll-memory behavior.
- Space membership and role checks.
- Global merge suggestion flow.
- Space merge navigation flow.
- API-specific save callbacks and toast/error messages.

Do not merge global person detail and space person detail in this spec. Global person detail is timeline-based, while space person detail is asset-grid based. Forcing those pages into one implementation would increase rebase risk without enough immediate reuse.

## Shared View Model

Shared components should operate on a small view model or simple prop contract rather than raw SDK DTOs. Route adapters map SDK data into the shared surface.

Core fields:

- `id`: person identifier.
- `displayName`: display-only name shown in headings, links, thumbnail titles, and viewer labels. For spaces, prefer `alias`, then `name`.
- `canonicalName`: underlying editable shared-space person name. Used by space editor rename controls and shown as secondary text when an alias is set.
- `thumbnailUrl`: URL generated by the adapter.
- `href`: link target generated by the adapter.
- `isHidden`: visibility state.
- `isFavorite`: optional; global people only for now.
- `type` and `species`: optional pet badge support.
- `assetCount` and `faceCount`: optional summary metadata.

Callbacks stay in adapters. The shared view model is data-only; route-specific behavior is passed through snippets, event handlers, or explicit modal save functions instead of being stored on the person object.

The shared components should not import global people routes or space routes. They should receive URLs, render snippets, and save functions from adapters.

## Global People Behavior

Global people should preserve existing behavior:

- Inline rename from the people grid.
- Hide from the grid action menu.
- Merge from the grid action menu using the existing merge suggestion/navigation behavior.
- Favorite/unfavorite from the grid action menu.
- Birth date editing through the existing person actions and modal.
- Existing search, scroll-memory, and websocket thumbnail update behavior.

Existing global components can remain as adapters:

- `people-infinite-scroll.svelte` delegates to the shared grid.
- `people-card.svelte` delegates to the shared person tile.
- `manage-people-visibility.svelte` delegates to the shared visibility modal and keeps `updatePeople` as its save adapter.

## Space People Behavior

Space people should preserve existing behavior and add the unused capabilities:

- Editors can rename inline in the space people grid.
- Editors can hide people from the grid action menu.
- Editors can start merge from the grid action menu.
- Viewers see read-only person names and no edit action menu.
- Space people do not get favorite/unfavorite UI because the current backend DTO/API does not support space-person favorites.
- Display-only grid labels and thumbnail titles prefer `alias`, then `name`.
- The editor inline rename field remains bound to canonical `name` and continues saving through `updateSpacePerson({ name })`; alias editing is intentionally kept on the detail page.

The space people grid page remains responsible for:

- `getSpacePeople` pagination.
- `withHidden` loading for the visibility modal.
- Space role checks via `SharedSpaceRole`.
- Navigation to the space person detail and merge flow.

The space visibility adapter delegates to the shared visibility modal and saves each change through `updateSpacePerson({ isHidden })`.

## Space Person Detail Behavior

The space person detail page should expose:

- Alias display.
- Alias edit.
- Alias clear.
- Birth date display.
- Birth date edit.
- Birth date clear.

Display rules:

- Page title and primary heading use `alias || name || space.name`.
- If `alias` and `name` are both present, show the canonical `name` as secondary text.
- If `birthDate` is present, display it with the same localized date formatting style used by global person detail.

Save rules:

- Non-empty alias saves with `setSpacePersonAlias({ id: space.id, personId: person.id, sharedSpacePersonAliasDto: { alias } })`.
- Empty alias clears with `deleteSpacePersonAlias`.
- Birth date saves with `updateSpacePerson({ birthDate })`.
- Birth date clears with `updateSpacePerson({ birthDate: '' })`, relying on the existing empty-string-to-null validation used by the space-person update DTO.

Permission rules:

- Any space member can set or clear their own alias because the existing alias endpoints require shared-space read membership.
- Birth date editing requires editor permission because it uses `updateSpacePerson`, which requires shared-space update permission.
- Viewers can see and edit their own alias, and can see birth date but cannot edit birth date.

## Error Handling

Keep error handling in adapters and route files, not in generic presentational components, except for the shared visibility modal's existing save lifecycle.

Messages:

- Space people loading: `spaces_error_loading_people`.
- Space merge: `spaces_error_merging_people`.
- Alias save or clear failure: `spaces_error_saving_alias`.
- Alias save success: `spaces_alias_saved`.
- Alias clear success: `spaces_alias_cleared`.
- Birth date save success: `date_of_birth_saved`.
- Birth date save failure: `errors.unable_to_save_date_of_birth`.
- Visibility save success/failure: preserve current visibility messages.

If a partial visibility save occurs, preserve current behavior: count successful and failed updates, warn on failures, and apply local visibility overrides only after the save lifecycle completes.

## Testing

Use focused frontend tests plus minimal backend regression coverage. Existing shared-space service tests cover alias authorization and persistence, but they do not currently cover shared-space birth-date update or clearing.

Add shared component tests:

- Shared grid renders links, thumbnails, footers, and loading state.
- Shared tile renders favorite/pet badges only when provided.
- Shared tile renders action slots only when enabled.
- Shared visibility modal preserves local overrides as more people are loaded.
- Shared visibility modal saves only changed people.
- Shared visibility modal uses adapter display names so space aliases are treated as named people.

Update global adapter tests:

- Existing global visibility tests still pass through the adapter.
- Existing people-card behavior still exposes hide, merge, birth date, and favorite actions.
- Existing people infinite scroll behavior still triggers load-more through the shared grid.

Update space adapter tests:

- Space people page still renders editor inline rename.
- Space people page editor inline rename saves canonical `name`, not `alias`.
- Space people page hides edit controls for viewers.
- Space people page routes merge and hide actions through the shared tile shell.
- Space people page display-only labels use `alias || name`.
- Space visibility tests still save through `updateSpacePerson`.
- Space visibility displays aliases when present.

Add space detail tests:

- Displays alias as primary name and canonical name as secondary text.
- Saves non-empty alias through `setSpacePersonAlias` with `sharedSpacePersonAliasDto: { alias }`.
- Clears alias through `deleteSpacePersonAlias`.
- Allows viewers to save and clear their own alias.
- Displays existing birth date using localized date formatting.
- Saves birth date through `updateSpacePerson({ birthDate })`.
- Clears birth date through `updateSpacePerson({ birthDate: '' })`.
- Shows birth date editing for editors.
- Hides birth date editing for viewers.

Add backend regression coverage:

- Shared-space service `updateSpacePerson` persists `birthDate` through `updatePerson` and returns the updated birth date in the mapped response.
- Shared-space controller or DTO validation maps `{ birthDate: '' }` to `{ birthDate: null }` for `PUT /shared-spaces/:id/people/:personId`.

## Rebase And Merge-Conflict Constraints

This is a fork rebasing on upstream. The implementation should minimize conflict risk:

- Add new shared files instead of moving existing files.
- Keep route paths and imports stable where possible.
- Keep global and space route-level data flow local.
- Avoid backend runtime/API, SDK, migration, and OpenAPI changes.
- Avoid changing unrelated i18n keys beyond any missing English strings required for the new UI.
- Keep adapter edits narrow and behavior-preserving.

## Out Of Scope

- Backend runtime/API changes.
- SDK regeneration.
- Space-person favorites.
- Full convergence of global person detail and space person detail.
- Reworking merge semantics.
- Reworking global people search.
- Broad visual redesign.
