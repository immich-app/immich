Immich S3 Architecture Plan

Existing Data: Your current S3 data will be preserved. The new approach reads existing s3Key and s3Bucket from the database. No data migration needed.

Code Source: What Goes Where

From Original Immich (Keep)

  • All file handling (StorageCore)
  • All job processing (metadata, thumbnails, video)
  • Local storage paths
  • Asset upload flow

From Your Fork (Keep)

  • S3StorageAdapter KEEP
  • S3StorageManager KEEP
  • Database migrations (s3Key, etc) KEEP
  • fly.toml structure KEEP

From Your Fork (Remove)

  • uploadToS3Sync() REMOVE
  • uploadThumbnailsToS3Inline() REMOVE
  • uploadEncodedVideoToS3Inline() REMOVE
  • S3 key derivation fallback REMOVE

Current Flow (Broken)

HTTP Upload
→
Sync S3 Upload
→
Delete Local
→
Queue Jobs
→
Workers: File Not Found!

Problem: Local file deleted before workers can process it.

New Flow (Simple)

HTTP Upload
→
Save to /tmp
→
Queue Jobs
→
Process Locally
→
S3 Upload Job
→
Done!

Solution: S3 upload happens AFTER all processing, as final step.

Step-by-Step Implementation

Step Action Files
1 Simplify fly.toml to unified process (web + workers) fly.toml
2 Remove sync S3 upload code asset-media.service.ts
3 Remove inline S3 upload code media.service.ts
4 Add S3UploadAsset job (final step) s3-storage.service.ts
5 Queue S3Upload after thumbnails complete media.service.ts
6 Add presigned URL redirect for downloads asset-media.service.ts
7 Test and deploy -

Existing Data Handling

Scenario What Happens
Asset already in S3 (storageBackend='s3') Downloads use presigned URL - works as before
Asset still local (storageBackend='local') Queue S3UploadAsset job to migrate to S3
New uploads Process locally → S3 upload as final step

No data loss. No migration script needed. Existing S3 files stay in S3.
